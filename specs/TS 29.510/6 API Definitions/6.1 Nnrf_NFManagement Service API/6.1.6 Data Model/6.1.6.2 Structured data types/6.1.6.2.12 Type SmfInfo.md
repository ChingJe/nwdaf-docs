<tr class="odd">
<td>sNssaiSmfInfoList</td>
<td>array(SnssaiSmfInfoItem)</td>
<td>M</td>
<td>1..N</td>
<td>List of parameters supported by the SMF per S-NSSAI (NOTE 1).</td>
</tr>
<tr class="even">
<td>taiList</td>
<td>array(Tai)</td>
<td>O</td>
<td>1..N</td>
<td>The list of TAIs the SMF can serve. It may contain one or more non-3GPP access TAIs. The absence of this attribute and the taiRangeList attribute indicate that the SMF can be selected for any TAI in the serving network.</td>
</tr>
<tr class="odd">
<td>taiRangeList</td>
<td>array(TaiRange)</td>
<td>O</td>
<td>1..N</td>
<td>The range of TAIs the SMF can serve. It may contain non-3GPP access TAIs. The absence of this attribute and the taiList attribute indicate that the SMF can be selected for any TAI in the serving network.</td>
</tr>
<tr class="even">
<td>pgwFqdn</td>
<td>Fqdn</td>
<td>O</td>
<td>0..1</td>
<td>The FQDN of the PGW if the SMF is a combined SMF/PGW-C.</td>
</tr>
<tr class="odd">
<td>pgwIpAddrList</td>
<td>array(IpAddr)</td>
<td>O</td>
<td>1..N</td>
<td><p>When present, this IE shall contain the PGW IP addresses of the combined SMF/PGW-C. Each PGW IP address shall be encoded as an IPv4 or an IPv6 address (i.e. not as an IPv6 prefix).</p>
<p>This IE allows the NF Service consumer to find the target combined SMF/PGW-C by PGW IP Address, e.g. when only PGW IP Address is available.</p></td>
</tr>
<tr class="even">
<td>accessType</td>
<td>array(AccessType)</td>
<td>C</td>
<td>1..2</td>
<td><p>If included, this IE shall contain the access type (3GPP_ACCESS and/or NON_3GPP_ACCESS) supported by the SMF.</p>
<p>If not included, it shall be assumed the both access types are supported.</p></td>
</tr>
<tr class="odd">
<td>priority</td>
<td>integer</td>
<td>O</td>
<td>0..1</td>
<td><p>Priority (relative to other NFs of the same type) in the range of 0-65535, to be used for NF selection for a service request matching the attributes of the SmfInfo; lower values indicate a higher priority.</p>
<p>The NRF may overwrite the received priority value when exposing an NFProfile with the Nnrf_NFDiscovery service.</p>
<p>Absence of this attribute equals to having the same smfInfo priority as the priority defined at NFProfile/NFService level.</p>
<p>(NOTE 2)</p></td>
</tr>
<tr class="even">
<td>vsmfSupportInd</td>
<td>boolean</td>
<td>O</td>
<td>0..1</td>
<td><p>This IE may be used by an SMF to explicitly indicate the support of V-SMF capability and its preference to be selected as V-SMF.</p>
<p>When present, this IE shall indicate whether the V-SMF capability are supported by the SMF:</p>
<p>- true: V-SMF capability supported by the SMF</p>
<p>- false: V-SMF capability not supported by the SMF.</p>
<p>Absence of this IE indicates the V-SMF capability support of the SMF is not specified.</p>
<p>(NOTE 3)</p></td>
</tr>
<tr class="odd">
<td>ismfSupportInd</td>
<td>boolean</td>
<td>O</td>
<td>0..1</td>
<td><p>This IE may be used by an SMF to explicitly indicate the support of I-SMF capability and its preference to be selected as I-SMF.</p>
<p>When present, this IE shall indicate whether the I-SMF capability are supported by the SMF:</p>
<p>- true: I-SMF capability supported by the SMF</p>
<p>- false: I-SMF capability not supported by the SMF.</p>
<p>Absence of this IE indicates the I-SMF capability support of the SMF is not specified.</p>
<p>(NOTE 3)</p></td>
</tr>
<tr class="even">
<td>pgwFqdnList</td>
<td>array(Fqdn)</td>
<td>O</td>
<td>1..N</td>
<td>When present, this attribute provides additional FQDNs to the FQDN indicated in the pgwFqdn attribute. The pgwFqdnList attribute may be present if the pgwFqdn attribute is present.</td>
</tr>
<tr class="odd">
<td>smfOnboardingCapability</td>
<td>boolean</td>
<td>O</td>
<td>0..1</td>
<td><p>When present, this IE indicates the SMF supports SNPN Onboarding capability and User Plane Remote Provisioning. This is used for the case of Onboarding of UEs for SNPNs (see 3GPP TS 23.501 [2], clauses 5.30.2.10 and 6.2.6.2).</p>
<p>- false (default): SMF does not support SNPN Onboarding;</p>
<p>- true: SMF supports SNPN Onboarding.</p>
<p>(NOTE 4)</p></td>
</tr>
<tr class="even">
<td>smfUPRPCapability</td>
<td>boolean</td>
<td>O</td>
<td>0..1</td>
<td><p>When present, this IE indicates the SMF supports User Plane Remote Provisioning (UPRP) capability. This is used for the case of Onboarding of UEs for SNPNs (see 3GPP TS 23.501 [2], clauses 5.30.2.10 and 6.2.6.2).</p>
<p>- false (default): SMF does not support UPRP;</p>
<p>- true: SMF supports UPRP.</p></td>
</tr>
<tr class="odd">
<td colspan="5"><p>NOTE 1: If this S-NSSAIs is present in the SmfInfo and in the NFprofile, the S-NSSAIs from the SmfInfo shall prevail.</p>
<p>NOTE 2: An SMF profile may e.g. contain multiple SmfInfo entries, with each entry containing a different list of TAIs and a different priority, to differentiate the priority to select the SMF based on the user location. The priority in SmfInfo applies between SMFs or SMF Services with the same priority.</p>
<p>NOTE 3: The IE should only be registered when the SMF is configured to be preferably selected as V-SMF/I-SMF.</p>
<p>NOTE 4: The IE is deprecated and replaced by smfUPRPCapability attribute.</p></td>
</tr>

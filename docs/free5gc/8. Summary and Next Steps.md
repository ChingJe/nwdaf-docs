## How free5GC Is Governed

free5GC is not just a software project; it is a coordinated community effort. To keep the project aligned with 3GPP standards and ensure consistent development, free5GC follows a structured governance model. This model defines how decisions are made, how work is reviewed, and how contributors grow into leadership roles.

---

## Technical Steering Committee (TSC)

The Technical Steering Committee (TSC) plays a central role. It consists of multiple members, with key advisors mainly coming from NYCU and Saviah Technologies. The TSC holds monthly meetings to review the project's development and ensure it remains aligned with its goals.

In addition to guiding direction, the TSC is responsible for defining the project roadmap. The project coordinators then communicate this roadmap and provide guidance to committers. This cycle repeats continuously, ensuring that each functional change is delivered in a consistent and structured way.

---

## Committers and Contributors

The free5GC project relies on structured development management. Contributors who demonstrate expertise and consistent involvement may be promoted to committers during TSC discussions.

Committers are responsible for:

* Tracking community issues reported on the free5GC forum and GitHub
* Conducting initial reviews of Pull Requests (PRs)

Regular development meetings bring together project coordinators, committers, and contributors. Project coordinators act as a bridge, providing technical guidance and ensuring that development work stays aligned with the project roadmap.

## Versioning

As previously mentioned, TSC members discuss the project roadmap during TSC meetings. When a milestone is completed, a new version is released.

free5GC follows software versioning for version management. Each version consists of three numbers in the following format.

```text
vX.Y.Z
```

## Version Assignment

The free5GC project follows a structured release workflow to ensure each version meets quality, stability, and interoperability standards. The Technical Steering Committee (TSC) defines the scope of each release during regular meetings. When the planned development work meets the defined criteria, the release process begins.

### Regression Testing

Confirms that new changes do not introduce unexpected issues in existing functionality. This step verifies that established features continue to work as intended.

### Integration Testing

Checks that the system’s components interact correctly. free5GC includes many independent network functions, so integration testing confirms that these components work together reliably.

### Unit Testing

Validates individual components in isolation to ensure correctness, reliability, and expected behavior before they are combined into the larger system.

Once testing is successfully completed, new version numbers are assigned to each Git repository following version control rules. These repositories are categorized as follows:

* **Network Functions:** AMF, SMF, UPF, PCF, UDR, UDM, etc.
* **Generic Packages:** util, ngap, nas, pfcp, etc.
* **Deployments:** free5gc, free5gc-compose, free5gc-helm, etc.

With this release process ensuring quality and consistency, contributors can be confident that new versions of free5GC meet the project's standards. The next step is to see how contributors can actively participate in free5GC development.

## How to Contribute

Contributing to the free5GC project is a valuable way to support the 5G open source ecosystem while gaining practical development experience. The project welcomes community contributions in the form of code, documentation, bug reports, and feature suggestions.

Before you contribute, it is helpful to follow a few key steps to ensure a smooth and successful process.

### Step 1: Familiarize Yourself with the Project

Read the [official free5GC documentation](https://free5gc.org/) and explore the [GitHub repository](https://github.com/free5gc/free5gc).

Review the project’s structure, coding style, and contribution guidelines.

### Step 2: Set Up Your Development Environment

Follow the installation and build instructions provided in the repository.

Make sure your local environment matches the recommended system configuration.

Run the system successfully before attempting to make changes.

### Step 3: Identify an Area to Contribute

Check the project’s issue tracker for open bugs or feature requests.

Look for issues labeled “good first issue” or “help wanted.”

Join discussions in the mailing list or project chat to find out what areas need support.

### Step 4: Fork and Work on the Code

Fork the repository to your personal GitHub account.

Create a new branch for your contribution.

Follow the coding and formatting standards used in the project.

### Step 5: Test Your Changes

Perform unit tests, integration tests, and regression tests where applicable.

Ensure your changes do not break existing functionality.

### Step 6: Submit a Pull Request

Provide a clear title and description of your changes.

Link related issues or background information.

Be prepared to revise your code based on reviewer feedback.
